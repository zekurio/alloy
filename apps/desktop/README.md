# Alloy Desktop

Alloy desktop app — a thin Electron shell that connects to a self-hosted Alloy
server and loads its web app, Jellyfin-desktop style.

## How it works

The app loads the Alloy web UI **live from the server origin** rather than
bundling it, so the page runs with real server-origin cookies.

**Login happens in the system browser.** Electron's Chromium doesn't ship the
WebAuthn UI layer, so passkeys (and provider-blocked embedded OAuth) can't run
inside the app. Instead the desktop uses an RFC 8252 loopback handshake: it
opens the system browser at `/api/auth/desktop/authorize`, the user signs in
there with full passkey/OAuth/password support, confirms the desktop link, the
server returns a one-time code to a temporary `127.0.0.1` listener, and the
desktop exchanges it with a PKCE-style verifier for a session it injects into
the main window's cookie jar (`src/main/browser-login.ts`,
`src/main/session.ts`). A still-valid stored session skips the handshake.

Two window surfaces with deliberately different trust levels:

- **Overlay** (`src/renderer`) — the bundled, trusted connect screen. The
  **only** window granted the privileged `window.alloyNative` bridge (via
  `src/preload/overlay.ts`).
- **Main** — loads the remote server origin with a narrow `window.alloyDesktop`
  bridge for desktop settings and recording controls. IPC handlers validate the
  calling window and connected origin before performing native actions.

The main window is also origin-pinned to the connected server, denies browser
permission requests by default, and only opens normal `http:`/`https:` URLs in
the system browser. HTTP Alloy servers are allowed only for loopback development;
remote servers must use HTTPS.

The desktop shell stores a bounded recent-server list in Electron's `userData`
directory. Successful connections move a server to the top, and the desktop
settings can reconnect to or forget saved servers through validated IPC.

The recording settings drive the bundled Rust sidecar, which talks to libobs over
stdio IPC from the Electron main process.

## Layout

```
src/
  main/      Electron main process (lifecycle, windows, server probe, IPC, state)
  preload/   contextBridge bridge — exposed only to the overlay
  renderer/  React connect screen (reuses alloy-ui)
  shared/    IPC channel names + payload types shared across the three
```

## Develop

```
pnpm --filter alloy-desktop dev    # run with hot reload
pnpm --filter alloy-desktop build  # bundle main/preload/renderer to out/
pnpm --filter alloy-desktop dist:linux # build Linux AppImage, deb, and tar.gz artifacts
pnpm --filter alloy-desktop dist:win   # build the unpacked Windows app
pnpm --filter alloy-desktop dist:win:installer # build an unsigned Windows NSIS installer
pnpm --filter alloy-desktop dist:all   # build Linux artifacts, then the unpacked Windows app
pnpm --filter alloy-desktop test   # unit-test desktop URL policy
```

From the repository root, the same packaging commands are available as
`pnpm desktop:dist:linux`, `pnpm desktop:dist:win`,
`pnpm desktop:dist:win:installer`, and `pnpm desktop:dist:all`. Artifacts are
written to `apps/desktop/release/`.

On Linux/WSL, `pnpm desktop:dist:win` builds `release/win-unpacked` and skips the
NSIS installer. The NSIS installer is built by GitHub Actions on Windows.

Windows distribution builds require a bundled OBS runtime. Set
`ALLOY_OBS_RUNTIME_DIR` to an OBS Studio runtime root, or to its `bin` or
`bin/64bit` directory, before running `pnpm desktop:dist:win` or
`pnpm desktop:dist:win:installer`. The runtime is accepted only when `obs.dll`
is present. Regular development builds still tolerate a missing bundled runtime
and can use a system OBS install.

## Release

Desktop releases use the app-local version in `apps/desktop/package.json` and
tags named `desktop-vX.Y.Z`. A desktop-only release publishes desktop
installers only; it does not publish Alloy server or machine learning images.

The first release target is an unsigned Windows x64 NSIS installer generated by
electron-builder under `apps/desktop/release/`. SmartScreen warnings are
expected until signing is added.

The desktop release workflow downloads a pinned OBS Studio portable runtime and
fails the release if the staged runtime does not contain `obs.dll`.
